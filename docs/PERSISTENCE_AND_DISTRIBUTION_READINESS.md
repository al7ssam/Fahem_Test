# جاهزية الاستمرارية والتوزيع — تحليل فقط (Phase E)

**هذا المستند ليس خطة تنفيذ Redis أو عقد متعددة.** هو قائمة تحقق للمخاطر والافتراضات إذا قرر الفريق لاحقاً **Phase 3 حقيقي** (HA، تعدد عقد، أو حالة مباراة خارج الذاكرة).

مرتبط: [عقد أحادية العقدة](RUNTIME_SINGLE_NODE_CONTRACT.md)، [ملكية المؤقتات](RUNTIME_TIMER_OWNERSHIP.md).

---

## 1) ما هو «عدائي للـ persistence» اليوم

| المنطقة | السبب |
|---------|--------|
| `GameManager` — جميع الـ `Map` للوبي/المقاعد/المقابس | لا مفتاح إصدار عملية؛ لا تكرار آمن عبر العقد |
| `Match` — حالة اللاعبين والجولة في الذاكرة | يختفي بالكامل عند `restart` |
| `resumeSecretsByParticipant` — `Buffer` عشوائي في الذاكرة | لا يُنسخ عبر العقد دون قناة سرية مشتركة |
| مؤقتات الجولة واللوبي | غير قابلة للاستئناف عبر عملية |
| `io.to(socketId)` كوجهة نقل | يفترض محلية محرك Socket.IO |
| لقطات `matchReconnectSnapshot` | للمزامنة بين عميل وخادم **حيّ**؛ ليست سجل تاريخي |

## 2) ما يمكن أن يبقى محلياً حتى مع توزيع لاحق (مثاليات)

- قراءة إعدادات ثابتة من DB مع TTL.
- تسجيل structured logs و`/health/realtime` كمراقبة لكل عقدة.
- منطق «عرض» لا يؤثر على سلطة المباراة.

---

## 3) ما يعيق التوزيع دون إعادة تصميم

1. **`getRunningMatch(matchId)` محلي دائماً** — يحتاج موجه جلسة أو shard معروف.
2. **أسرار الاستئناف في الذاكرة فقط** — تحتاج تخزيناً آمناً قصير العمر أو سياق جلسة موقّع.
3. **«آخر مقبس يفوز» عبر `disconnect` لمقبس سابق** — يحتاج إما stickiness نقل أو إبطال جلسة مركزي.
4. **مؤقتات بدون مالك عبر العمليات** — تحتاج ساعة/جدولة خارجية أو leader واحد للمباراة.
5. **غياب `match` generation / epoch** — لمنع لقطات قديمة تُطبَّق على حالة جديدة بعد إعادة استخدام معرف.

## 4) إشارات تشغيلية قبل التفكير جدياً في Redis/متعدد العقد

راقب (من السجلات و`/health/realtime` و`stats`):

- اتجاه **فشل** `resume_match` غير المفسَّر (غير `bad_token` المتوقع بعد النشر).
- **`server_draining`** أثناء كل نشر (متوقع قصيراً) — إن طال، راجع ميزانية الإيقاف.
- **زمن إيقاف** يقترب من حدود الميزانية باستمرار.
- **`rate_limited`** مرتفع لجزء كبير من المستخدمين الأصحاء (قد يدل على عميل خاطئ أو هجوم).
- **`activeMatches`** يتذبذب بشكل لا يطابق أنماط اللعب (تسرب توجيه نادر).

---

## 5) معايير تبرّر بدء Phase 3 (تنفيذ بنية)

يجب توفر **كل** مما يلي على الأقل:

1. **SLA منتج** صريح لحالة المباراة عبر إعادة التشغيل/العقد (مثلاً: «قبول فقدان الجولة» مقابل «استئناف مدفوع»).
2. **حاجة سعة مثبتة** — مقبس واحد أو CPU واحد يصبح حجر الزاوية بقياسات، لا مجرد «نخاف من المستقبل».
3. **نموذج بيانات للمباراة الحية** — من يكتب، من يقرأ، TTL، وتضارب الكتابة.
4. **خطة اختبار انحدار** تشمل reconnect وعدّ تنازلي لوبي وغرفة خاصة **على بيئة متعددة العقد** أو محاكاة التوزيع.

حتى ذلك الحين: **الإبقاء على أحادية العقدة** أقل تكلفة تشغيلية وأقل سطح هجوم للتعقيد.
